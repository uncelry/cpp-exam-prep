## **What are the Rule of Three and the Rule of Five? Which functions must be consistent (copy/move constructors, assignment operators, destructor)? When must they be user-defined?**

## **Что такое Rule of Three и Rule of Five? Какие функции должны быть согласованы (конструкторы копирования/перемещения, операторы присваивания, деструктор)? Когда их нужно определять вручную?**

---

# 1️⃣ Проблема, которую решают эти правила

В C++ объект может **владеть ресурсами**:

- динамической памятью (`new`)
    
- файловыми дескрипторами
    
- сокетами
    
- mutex
    
- другими системными ресурсами
    

Если объект копируется или уничтожается **неправильно**, возникают:

- double free
    
- утечки памяти
    
- dangling pointers
    

Поэтому специальные функции класса должны быть **согласованы**.

---

# 2️⃣ Специальные функции класса (special member functions)

В C++ есть **6 специальных функций**:

1️⃣ **Destructor**

```cpp
~Class();
```

2️⃣ **Copy constructor**

```cpp
Class(const Class&);
```

3️⃣ **Copy assignment operator**

```cpp
Class& operator=(const Class&);
```

4️⃣ **Move constructor** (C++11)

```cpp
Class(Class&&);
```

5️⃣ **Move assignment operator**

```cpp
Class& operator=(Class&&);
```

6️⃣ **Default constructor**

Но Rule of Three/Five касается **первых пяти**.

---

# 3️⃣ Rule of Three

## 📌 Формулировка

**Если класс требует пользовательской реализации одного из этих трёх:**

- деструктора
    
- copy constructor
    
- copy assignment operator
    

**то почти наверняка нужно реализовать все три.**

---

## Почему?

Потому что объект управляет ресурсом.

Если вы написали деструктор — значит:

- ресурс освобождается вручную
    
- значит копирование должно быть **глубоким**
    

---

# 4️⃣ Пример проблемы без Rule of Three

```cpp
class Buffer {
private:
    int* data;

public:
    Buffer(int size) {
        data = new int[size];
    }

    ~Buffer() {
        delete[] data;
    }
};
```

Если сделать копию:

```cpp
Buffer a(10);
Buffer b = a;
```

Происходит **shallow copy**:

```id="tpicv4"
a.data → память
b.data → та же память
```

Когда объекты уничтожаются:

```id="l6p7ps"
delete[] data
delete[] data
```

→ **double free**

---

# 5️⃣ Правильная реализация (Rule of Three)

```cpp
class Buffer {
private:
    int* data;
    int size;

public:
    Buffer(int s) : size(s) {
        data = new int[size];
    }

    ~Buffer() {
        delete[] data;
    }

    Buffer(const Buffer& other) {
        size = other.size;
        data = new int[size];

        std::copy(other.data, other.data + size, data);
    }

    Buffer& operator=(const Buffer& other) {
        if (this == &other)
            return *this;

        delete[] data;

        size = other.size;
        data = new int[size];

        std::copy(other.data, other.data + size, data);

        return *this;
    }
};
```

Теперь копирование **глубокое**.

---

# 6️⃣ Rule of Five

С появлением **move semantics (C++11)** правило расширилось.

## 📌 Формулировка

Если класс реализует один из:

- destructor
    
- copy constructor
    
- copy assignment
    

то, скорее всего, нужно реализовать **ещё и move операции**.

---

Теперь полный набор:

1️⃣ Destructor  
2️⃣ Copy constructor  
3️⃣ Copy assignment  
4️⃣ Move constructor  
5️⃣ Move assignment

---

# 7️⃣ Move constructor

Позволяет **передать ресурс**, а не копировать его.

```cpp
Buffer(Buffer&& other) noexcept {
    data = other.data;
    size = other.size;

    other.data = nullptr;
    other.size = 0;
}
```

---

# 8️⃣ Move assignment

```cpp
Buffer& operator=(Buffer&& other) noexcept {
    if (this == &other)
        return *this;

    delete[] data;

    data = other.data;
    size = other.size;

    other.data = nullptr;
    other.size = 0;

    return *this;
}
```

---

# 9️⃣ Что происходит при move

```cpp
Buffer a(100);

Buffer b = std::move(a);
```

Теперь:

```id="rc2oh0"
b.data → память
a.data → nullptr
```

Копирования нет.

---

# 🔟 Когда функции генерируются автоматически

Компилятор может автоматически сгенерировать:

- destructor
    
- copy constructor
    
- copy assignment
    
- move constructor
    
- move assignment
    

Но только если это безопасно.

---

# 1️⃣1️⃣ Когда нужно писать их вручную

Нужно **user-defined**, если класс:

### Управляет ресурсами

Например:

```cpp
int* ptr;
FILE*
socket
mutex
```

---

### Или владеет raw pointer

```cpp
class A {
    int* data;
};
```

---

### Или нужен глубокий copy

---

# 1️⃣2️⃣ Когда писать не нужно

Если используются **RAII объекты STL**:

```cpp
std::vector
std::string
std::unique_ptr
std::shared_ptr
```

Компилятор всё сделает правильно.

Это называется:

## **Rule of Zero**

---

# 1️⃣3️⃣ Rule of Zero

Лучший стиль C++:

> Не писать специальные функции вообще.

Пример:

```cpp
class A {
private:
    std::vector<int> data;
};
```

Никаких проблем с памятью.

---

# 1️⃣4️⃣ Почему функции должны быть согласованы

Потому что они работают с **одним и тем же ресурсом**.

Если:

- copy делает глубокую копию
    
- destructor освобождает память
    

то move должен **переносить владение**.

---

# 1️⃣5️⃣ Частые ошибки студентов

❌ Реализуют destructor, но не copy constructor  
❌ Делают shallow copy  
❌ Забывают проверку self-assignment

```cpp
if (this == &other)
```

❌ Не обнуляют moved-from объект

---

# 1️⃣6️⃣ Минимальный экзаменационный пример

```cpp
class String {
private:
    char* data;

public:
    String(const char* s) {
        data = strdup(s);
    }

    ~String() {
        free(data);
    }

    String(const String& other) {
        data = strdup(other.data);
    }

    String& operator=(const String& other) {
        if (this != &other) {
            free(data);
            data = strdup(other.data);
        }
        return *this;
    }
};
```

---

# 1️⃣7️⃣ Главное, что студент должен сказать на экзамене

**Rule of Three**

Если класс реализует:

- destructor
    
- copy constructor
    
- copy assignment
    

то нужно реализовать **все три**.

---

**Rule of Five**

В C++11 добавились:

- move constructor
    
- move assignment
    

Поэтому нужно реализовать **пять функций**.

---

**Когда их нужно писать:**

- если класс владеет ресурсами
    
- если используется raw pointer
    
- если нужен глубокий copy
    

---

**Лучший стиль:**

Использовать **Rule of Zero** и RAII-классы STL.
